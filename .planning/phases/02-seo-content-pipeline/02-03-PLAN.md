  <action>
    1. Create `src/lib/seo/cluster-resolver.ts`:

    2. Define and export `ClusterContext` interface:
       ```typescript
       export interface ClusterContext {
         clusterId: string
         role: 'pillar' | 'satellite'
         pillar: { slug: string; title: string; url: string }
         siblings: Array<{ slug: string; title: string; url: string; role: string }>
       }
       ```

    3. Export `async function resolveCluster(page: WagtailPageWithSEO): Promise<ClusterContext | null>`:
       - If page.cluster_id is falsy, return null (page has no cluster assignment)
       - Fetch all pages with the same cluster_id using `getPages` from wagtail client. The Wagtail API supports filtering by field value, so use params: `{ cluster_id: page.cluster_id }`. BUT since Wagtail v2 API filtering requires the field to be configured as a filter -- and we can't guarantee that at this point -- use an alternative: fetch ALL pages of all types and filter client-side by cluster_id. This is acceptable because the site has <100 pages total and results are ISR-cached.

       ACTUALLY, the better approach: Use a dedicated API call. Since Wagtail pages API supports `?fields=*` and we can filter by `search=` or by `type=`, the simplest reliable approach is:
       - Call `getPages` with NO type filter (fetch all page types) but add a `search` param or just fetch all pages with `fields=cluster_id,cluster_role,pillar_slug,title` and filter client-side
       - For efficiency, use a simpler approach: Accept the page's cluster_id, then make ONE API call to get all pages, filter by cluster_id in JS
       - Cache the result with a reasonable revalidate (3600s, same as default)

       Alternative simpler approach that avoids large fetches:
       - The resolver receives the CURRENT page (already fetched with cluster data)
       - If the page is a satellite, fetch the pillar by slug using `getPageBySlug` with the page's `pillar_slug` field
       - Then fetch all pages with `search` or list them from a known type

       RECOMMENDED approach (pragmatic):
       - Create a helper `getClusterPages(clusterId: string)` that calls the Wagtail pages API with `fields=title,cluster_id,cluster_role,pillar_slug` as a single batch call
       - Filter the results client-side by cluster_id match
       - Separate into pillar (cluster_role === 'pillar') and satellites
       - Build ClusterContext from the filtered results
       - If no pillar found in results and current page IS the pillar, use current page as pillar

    4. Handle edge cases:
       - Page has cluster_id but no other pages share it yet -> return ClusterContext with empty siblings
       - Cluster has no pillar page -> return null (invalid cluster configuration)
       - API error -> return null (don't break the page)

    5. Update `src/lib/seo/index.ts` to re-export resolveCluster and ClusterContext

    Use FetchConfig with tags: ['cluster', clusterId] so cluster data can be revalidated when content changes.
