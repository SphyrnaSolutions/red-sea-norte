Output: /api/revalidate POST endpoint secured with a shared secret, plus a checkpoint for configuring the Wagtail webhook.
</objective>

<task type="auto">
  <name>Task 1: Create /api/revalidate POST endpoint for on-demand ISR</name>
  <files>src/app/api/revalidate/route.ts</files>
  <action>
    Create POST handler at src/app/api/revalidate/route.ts:

    1. Accept JSON body with:
       - `secret`: string (must match REVALIDATION_SECRET env var)
       - `type`: string (content type slug: "blog", "rutas", "experiencias", "ofertas", "cursos", "homepage")
       - `slug`: optional string (specific page slug for targeted revalidation)

    2. Validate secret against process.env.REVALIDATION_SECRET:
       - If missing or mismatch: return 401 { error: "Invalid secret" }
       - If REVALIDATION_SECRET env var not set: return 500 { error: "Revalidation not configured" }

    3. Call revalidateTag() from 'next/cache':
       - Always revalidate the type tag (e.g., "blog", "blog-list")
       - If slug provided, also revalidate the specific tag (e.g., "blog-ruta-norte")
       - Tag names must match the tags used in data layer fetchers:
         - blog: tags ['blog', 'blog-list'] and ['blog', 'blog-{slug}']
         - rutas: tags ['rutas', 'rutas-list'] and ['rutas', 'ruta-{slug}']
         - experiencias: tags ['experiencias', 'experiencias-list'] and ['experiencia-{slug}']
         - ofertas: tags ['ofertas', 'ofertas-list'] and ['oferta-{slug}']
         - cursos: tags ['cursos', 'cursos-list'] and ['curso-{slug}']
         - homepage: revalidateTag('homepage')

    4. Return 200 { revalidated: true, type, slug? }

    5. Add try/catch with 500 response on unexpected errors.

    6. Do NOT apply rate limiting to this endpoint -- it's server-to-server with secret auth.

    7. Add REVALIDATION_SECRET to .env.example with a comment explaining its purpose.
  </action>
