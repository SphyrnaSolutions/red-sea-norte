    **ArticleHero.tsx:**
    - Props: title (string), heroImage (string), heroAlt (string), excerpt (string), publishedAt (string optional), author (AuthorData optional)
    - Renders a server-side `<h1>` with the page title -- this is critical for SEO. Use design tokens: --font-display for the H1, font size from globals.css H1 size (72px desktop, scale down mobile)

    **ArticleBody.tsx:**
    - Props: blocks (Block[])
    - Imports `BlockRenderer` from `@/components/blocks`
    - Maps over blocks, rendering each through BlockRenderer
    - Enforces heading hierarchy: the HeadingBlock should already handle H2/H3/H4. The body renderer should NOT allow H1 in body blocks. If a heading block has level=1, render as H2 instead (safety net).
    - Wraps content in a max-width container (max-w-[800px] per existing blog pattern) centered on page

    **InterlinksSection.tsx:**
    - Props: relatedPages (RelatedPage[]), title (string, default "Contenido Relacionado")
    - Renders a grid of related content cards (2 cols desktop, 1 mobile)
    - Each card: linked image thumbnail, title as link, short excerpt
    - Uses next/link for internal navigation
    - If relatedPages is empty, render nothing (return null)
    - Styling: Section background using --color-ocean-deep at 3% opacity, cards with subtle border and hover shadow

    **ClusterNavigation.tsx:**
    - Props: clusterId (string), currentSlug (string), clusterRole ('pillar' | 'satellite'), pillarSlug (string)
    - This component will consume Phase 2's cluster resolver to show sibling/satellite pages
    - For now, create the component shell that accepts these props and renders a placeholder section with the label "Explora esta ruta"
    - Add a TODO comment: "// TODO: Connect to Phase 2 cluster resolver when available"
    - When connected, it should render: link to pillar page (if current is satellite), list of sibling pages in the cluster

    **TableOfContents.tsx:**
    - Props: blocks (Block[])
    - Scans body blocks for heading types (H2, H3) and generates an ordered list of anchor links
    - Each heading gets an auto-generated id (slugified heading text)
    - Renders as a sticky sidebar on desktop (hidden on mobile or shown as collapsible)
    - If no headings found, render nothing
    - Styling: Use --font-mono for ToC text, --color-ocean-deep for links

    **SEOArticleTemplate.tsx:**
    - Props: page (SEOArticlePageData)
    - Composition order (this IS the template structure):

      3. `<article>` wrapper
      4. ArticleHero (title as H1, hero image, excerpt)
      5. Two-column layout on desktop: TableOfContents (sticky left) + ArticleBody (main right). Single column on mobile with ToC at top.
      6. InterlinksSection (related content from cluster engine)
      7. ClusterNavigation (cluster sibling/satellite nav)
      8. Lead capture CTA: Import InlineLeadSection from `@/components/organisms/InlineLeadSection` and render at bottom
    - The template must use semantic HTML: `<article>`, `<header>`, `<nav>`, `<aside>`, `<section>`
    - All styling via Tailwind + design tokens from globals.css. No raw hex colors.
    - Named export: `export function SEOArticleTemplate({ page }: { page: SEOArticlePageData })`
