<objective>
Rewrite the Wagtail API client from axios to native fetch with Next.js ISR caching, add pagination support for >20 items, and make fallback mode production-safe.

Purpose: The current axios-based client bypasses Next.js per-request caching. Without pagination, content types with >20 items silently lose entries. The fallback config defaults to serving mock data in production.

Output: A native fetch client with ISR caching + pagination, updated fetchers that accept cache config, and a production-safe fallback configuration.
</objective>

  <name>Task 1: Rewrite Wagtail client from axios to native fetch with pagination</name>
  <files>src/lib/wagtail/client.ts, src/lib/wagtail/types.ts</files>
  <action>
Rewrite `src/lib/wagtail/client.ts` completely:

1. Remove all axios imports and the AxiosInstance/interceptor code.

2. Create a `FetchConfig` interface with `revalidate?: number | false` and `tags?: string[]`. Export it.

3. Create `wagtailFetch<T>(endpoint, params?, config?)` function:
   - Build URL from `WAGTAIL_API_URL` + endpoint + query params
   - Use native `fetch()` with `next: { revalidate: config.revalidate ?? 3600, tags: config.tags ?? [] }`
   - Add `Content-Type: application/json` header
   - Log requests in development: `console.log('[Wagtail API] GET', url.pathname + url.search)`
   - On non-ok response, throw `Error('Wagtail API error: {status} {statusText} for {endpoint}')`
   - Return `response.json()` typed as `T`
   - Keep 10000ms timeout via `AbortSignal.timeout(10000)`

4. Rewrite `getPages<T>(type, params?, config?)`:
   - Add `config?: FetchConfig` parameter
   - Implement offset-based pagination loop with `BATCH_SIZE = 20`
   - Loop: fetch with `limit: BATCH_SIZE, offset: N`, accumulate items, break when `allItems.length >= response.meta.total_count`
   - Use immutable pattern: `allItems = [...allItems, ...response.items]`
   - Error handling: let errors propagate (no try/catch that swallows -- the data layer handles errors)

5. Rewrite `getPageBySlug<T>(type, slug, config?)`:
   - Add `config?: FetchConfig` parameter
   - Use `wagtailFetch` instead of axios
   - Return `response.items[0] || null`
   - On 404-equivalent (empty items), return null
   - Let other errors propagate

6. Rewrite `getPage<T>(id, config?)`, `getImage(id, config?)`, `searchPages<T>(query, type?, config?)`, `getPreviewPage<T>(contentType, token)`:
   - Same pattern: replace axios with `wagtailFetch`, add optional `config` param where appropriate
   - `getPreviewPage` should NOT cache (set `revalidate: 0` or `cache: 'no-store'`) since previews must be fresh
   - Keep the same return type contracts (null for not-found, throw for errors)

7. Update `WagtailPageMeta` in `src/lib/wagtail/types.ts`:
   - Add `first_published_at: string` to the WagtailPageMeta interface (move from client.ts if it's there, or add)
   - Add optional `last_published_at?: string` for future use
   - Keep existing WagtailPageMeta in client.ts in sync, or better: move the interface to types.ts and import from there to avoid duplication

8. Export `FetchConfig` and `WagtailAPIResponse` types from client.ts for use by fetchers.

IMPORTANT: Do NOT import axios anywhere. The entire point is to use native fetch for Next.js ISR compatibility.

PITFALL: In Next.js 16, fetch defaults to NO cache. Every call MUST explicitly set `next: { revalidate: N }` to opt into ISR.
